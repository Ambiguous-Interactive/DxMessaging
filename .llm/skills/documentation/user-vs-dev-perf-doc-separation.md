# User vs Dev Perf Doc Separation

> **One-line summary**: The USER page `docs/architecture/performance.md` carries
> headline numbers, the cross-library comparison matrix, scope labels, the "what
> this means" notes, and the single AUTOGENERATED marker pair the renderer owns;
> DEV/ops content lives in `docs/runbooks/perf-benchmark-methodology.md`.

## Overview

Performance documentation serves two readers with opposite needs. A user wants
the current numbers and a short read on what they mean. A maintainer wants the
methodology, the CI mechanics, baseline capture, the regression gate, and the
how-to for bumping a comparison library. DxMessaging keeps these on two pages so
neither dilutes the other.

The user page is `docs/architecture/performance.md`. The developer/ops page is
`docs/runbooks/perf-benchmark-methodology.md`. The user page links to the
runbook for depth; the runbook links back to the user page for the rendered
tables.

## Problem Statement

When one page tries to serve both readers, two failures follow:

- **The user page bloats.** Methodology, baseline capture, and gate internals
  push the numbers below the fold and bury the short read a user needs.
- **The numbers and the deep content fight for the same region.** The renderer
  rewrites a marked region in place; mixing hand-maintained methodology into
  that region risks the auto-commit overwriting prose or the prose breaking the
  marker pair.

## Solution

### What lives on the USER page

`docs/architecture/performance.md` carries only user-facing content:

- Headline throughput numbers and the cross-library comparison matrix
  (throughput plus memory).
- Scope labels (Standalone, PlayMode) and which scope is the headline.
- A "how to read these tables" / "what this means" section.
- Exactly one AUTOGENERATED marker pair that the renderer owns.

The renderer rewrites only the region between the markers:

```text
<!-- AUTOGENERATED:DISPATCH-THROUGHPUT BEGIN -->
... renderer-owned tables and matrices ...
<!-- AUTOGENERATED:DISPATCH-THROUGHPUT END -->
```

`scripts/unity/render-perf-doc.js` requires exactly one such pair and fails if
the markers are missing or duplicated. Keep the pair intact and do not hand-edit
inside it.

### What lives on the DEV/ops page

`docs/runbooks/perf-benchmark-methodology.md` carries everything a maintainer
needs and a user does not:

- Measurement methodology internals (the single window, the GC sampling).
- Build and runtime configuration per leg.
- Baseline capture and the opt-in regression smoke gate.
- The hot-path PR rule.
- The comparison-package single-source how-to.

## Where Content Goes

| Content                                 | User page | Dev runbook |
| --------------------------------------- | --------- | ----------- |
| Headline numbers, comparison matrix     | Yes       | No          |
| Scope labels, "what this means"         | Yes       | No          |
| AUTOGENERATED marker pair               | Yes       | No          |
| Methodology internals, GC sampling      | No        | Yes         |
| Baseline capture, regression gate       | No        | Yes         |
| Hot-path PR rule, package single-source | No        | Yes         |

## Common Pitfalls

- "I will add the methodology to the user page so it is all in one place." That
  bloats the user page; link to the runbook instead.
- "I will hand-edit a number inside the marker pair." The renderer overwrites
  it; edit the source data and re-render.
- "I will add a second marker pair for the memory table." The renderer owns one
  pair and fails on duplicates; keep a single pair.
- "I will move the comparison matrix to the runbook." The matrix is user-facing
  and belongs on the user page; the runbook holds the how-to for bumping a
  library.
